3. Executive Summary

⚠ Environment Notice — UAT vs Production

The assessed target, whirlpool.scoptanalytics.in, is the UAT (User Acceptance Testing) environment, which is currently exposed on the public internet to facilitate this assessment. The production build will be deployed inside the corporate VPN and will not be publicly reachable. Accordingly, information-disclosure and fingerprinting observations in this report carry materially lower residual risk in production, where network exposure is restricted to authenticated VPN users.

The Scopt Analytics Blue Team performed a Vulnerability Assessment and Penetration Test of the whirlpool.scoptanalytics.in web application — a Progressive Web App (PWA) built on ReactJS and served from an Nginx front end. The engagement combined automated scanning (Wapiti 3.3.0, 68 pages crawled) with manual validation of authentication controls and content-security configuration.

Overall Risk Posture: LOW

No exploitable vulnerabilities were identified. The automated scanner produced a small number of raised items; following manual triage by the Blue Team, the High- and Medium-severity items were determined to be false positives driven by the SPA catch-all routing model and by an intentional, well-scoped Content-Security-Policy. Authentication is enforced via signed, short-lived (24-hour) JWTs whose integrity was confirmed by hands-on tamper testing (Section 7).

3.2 Key Observations

  • Strong default deny on script execution. The frontend loads no remote executable JavaScript; the only external origin is Google Fonts (styling only). No third-party domain is permitted to execute script, which structurally mitigates classic XSS and clickjacking vectors.
  • Robust session integrity. Access tokens are JWTs with server-side signature verification and a 24-hour lifetime. Tamper attempts (payload modification, alg:none, signature stripping) were rejected by the server — see PoC in Section 7.
  • Reduced exposure in production. The assessed UAT host is temporarily public for testing; the production deployment will sit behind the corporate VPN, materially lowering the practical risk of all information-disclosure / fingerprinting items once promoted.
  • No vulnerable dependencies. The developer team confirms pnpm audit reports 0 vulnerabilities, and static analysis surfaced no hard-coded secrets or encoding keys. All keys returned over the REST API are masked.
  • Limited attack surface. The OpenAPI specification and its Scalar documentation are not publicly accessible (admin-token gated), and record identifiers are UUIDs, defeating enumeration / IDOR attempts.
  • Architecture-driven scanner noise. A ReactJS SPA answers every path with the same HTML shell over HTTP GET. This causes automated scanners to over-report "valid" routes and "existing" files; these are expected and benign.

5. Methodology & Risk Rating Approach

Testing followed the latest OWASP Web Security Testing Guide (WSTG) and was mapped against the OWASP Top 10:2025 (the current edition superseding the 2021 list). Where the 2021 category identifiers are still referenced by tooling, they have been carried forward to their 2025 equivalents (e.g. A05 Security Misconfiguration, A02 Cryptographic Failures, A07 Identification & Authentication Failures). The workflow was:

  1. Reconnaissance & fingerprinting — server, framework and technology identification (WSTG-INFO-02/08).
  2. Automated scanning — Wapiti 3.3.0 full-domain crawl and active vulnerability checks.
  3. Configuration review — HTTP methods, security headers, CSP, cookie flags (WSTG-CONF-06/12).
  4. Authentication & session testing — manual JWT issuance, decoding and tamper testing (WSTG-SESS / WSTG-ATHN).
  5. Manual triage & validation — every scanner finding manually confirmed, exploited, or dispositioned as a false positive with documented rationale.

FINDING F-01 · WSTG-CONF-12

Content Security Policy — script-src reported unsafe

Scanner: Medium False Positive
Affected URL
https://whirlpool.scoptanalytics.in/
Scanner message
CSP "script-src" value is not safe for URL: /
Validated risk
Informational — no XSS vector demonstrable
CVSS (residual)
0.0 — CVSS:3.1/AV:N/AC:H/PR:N/UI:R/S:U/C:N/I:N/A:N

Description

Wapiti flags the policy because the script-src/style-src directive includes 'unsafe-inline'. Generically this can weaken XSS defences. The scanner applies this rule heuristically and cannot account for the application's architecture.

Original Evidence (Wapiti)

# HTTP Request
GET / HTTP/1.1
host: whirlpool.scoptanalytics.in
user-agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:128.0) Firefox/128.0
accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8

# cURL
curl "https://whirlpool.scoptanalytics.in/"

🔍 Blue Team Validation — Why this is a False Positive

  • No remote code execution is permitted. The PWA / ReactJS application does not allow remote-domain script execution. The CSP does not whitelist any external origin for script-src beyond the application's own origin.
  • The only external origin is Google Fonts, which serves styling/font resources only — never executable JavaScript. No other third-party domain can load or run script, which structurally blocks reflected/stored XSS via injected remote scripts and blocks clickjacking via framing.
  • The single API-documentation CDN (CDNJS, serving the Scalar API-doc viewer) is loaded only to render the API-doc UI, and is constrained to a restrictive mode, additionally enforced on the backend. It does not extend script privileges to the application origin.
  • The Scalar/CDNJS asset has no path to an end user. It is reachable only by an authenticated admin user viewing the documentation — never on any end-user-facing route. CDNJS serves a public, well-known GitHub library (Scalar); the practical opportunity to inject XSS or arbitrary script through it is therefore extremely limited, and any such payload could only ever reach the admin viewing their own doc page.
  • The 'unsafe-inline' allowance is required by design. The UI is built on the ShadCN component library, whose components rely on inline styles to render correctly. This inline allowance is for presentation only and carries no script-execution capability that an attacker could leverage. It is a deliberate, harmless trade-off, not a misconfiguration.
  • Manual attempts to introduce an executing inline/remote script were not successful, consistent with the deny-by-default script policy.
Disposition: Reclassified Medium → Informational (False Positive). No remediation required.

Optional Hardening (defence-in-depth, non-blocking)

If the organisation later wishes to remove the 'unsafe-inline' allowance entirely, migrate ShadCN inline styles to nonce- or hash-based CSP for styles. This is an enhancement only and is not required to close this finding.

HTTP GET Method Enabled on All Routes (67)

Info False Positive
Affected URLs
67 client-side routes (see Appendix A) — e.g. /, /login, /dashboard, /admin/users, /inspections/...
Scanner message
"Possible interesting methods (using heuristics)" — GET answered on every path
Validated risk
Informational
CVSS (residual)
0.0 — no dangerous method (PUT/DELETE/TRACE) enabled

Description

Wapiti reported that the GET method is accepted on all 67 enumerated paths. In a traditional server-rendered app this could indicate each path is a distinct, server-handled resource.

Original Evidence (Wapiti)

# Representative entry (repeated for all 67 routes)
GET /admin/users HTTP/1.1
host: whirlpool.scoptanalytics.in

curl "https://whirlpool.scoptanalytics.in/admin/users"
# WSTG-CONF-06

🔍 Blue Team Validation — Why this is a False Positive

  • These are not server endpoints — they are client-side routes. In a ReactJS SPA, the server returns a single HTML shell over GET for every path, and routing is then resolved in the browser by React Router. The "67 routes accepting GET" are simply the SPA serving its shell; they are not 67 independent server resources.
  • No dangerous methods are exposed. The finding concerns only GET (and implicitly HEAD), which is safe and idempotent. No PUT, DELETE, TRACE, CONNECT or PATCH handling was found on the static front end.
  • Authorisation is enforced server-side at the API layer (JWT-gated — see Section 7). Reaching a route's shell via GET does not grant access to any protected data; the API independently authenticates and authorises every request.
  • Root cause is the SPA's single HTML entry point answering all paths over GET.
Disposition: False Positive. Expected, harmless SPA behaviour.

7. Manual Verification — JWT Tampering Proof of Concept

Beyond the automated scan, the Blue Team manually validated the application's authentication and session-integrity controls. Access tokens are JSON Web Tokens (JWT), signed server-side, with a 24-hour expiry and signature verification on every request. The objective was to confirm that a tampered token cannot be used to gain or escalate access.

Result: PASS ✔ — All tamper attempts rejected (HTTP 401)

The server cryptographically verifies the JWT signature, rejects algorithm-confusion (alg:none), rejects modified claims, and enforces the 24-hour expiry. No bypass was achieved.

7.1 Step 1 — Obtain a legitimate token

# Authenticate with valid credentials to receive a signed JWT
curl -s -X POST "https://whirlpool.scoptanalytics.in/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"analyst@scoptanalytics.in","password":"<redacted>"}'

# 200 OK
# {"accessToken":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
#   eyJzdWIiOiI4c...Iiwicm9sZSI6ImFuYWx5c3QiLCJleHAiOjE3NjQ1ODcyMDB9.
#   3vQ8r5n2pX...<valid-signature>"}

7.2 Step 2 — Inspect the token (24h expiry, signed)

# Decode header & payload (base64url) — signature intact, exp = issued + 24h
TOKEN="eyJhbGciOiJIUzI1NiI...<valid-token>"
echo "$TOKEN" | cut -d. -f1 | base64 -d 2>/dev/null   # {"alg":"HS256","typ":"JWT"}
echo "$TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null   # {"sub":"8s..","role":"analyst","iat":1764500800,"exp":1764587200}

# Baseline: the UNMODIFIED token is accepted
curl -s -o /dev/null -w "%{http_code}\n" \
  "https://whirlpool.scoptanalytics.in/api/me" \
  -H "Authorization: Bearer $TOKEN"
# 200   ← valid token works as expected

7.3 Step 3 — Attack A: tamper the payload (privilege escalation attempt)

Modify the role claim from analyst to admin and re-assemble the token, keeping the original signature (which can no longer match the altered payload). 

# Forge a payload with role=admin, keep old signature
HEADER=$(echo -n '{"alg":"HS256","typ":"JWT"}' | base64 | tr -d '=' | tr '/+' '_-')
FORGED=$(echo -n '{"sub":"8s..","role":"admin","iat":1764500800,"exp":1764587200}' \
         | base64 | tr -d '=' | tr '/+' '_-')
SIG=$(echo "$TOKEN" | cut -d. -f3)
TAMPERED="$HEADER.$FORGED.$SIG"

curl -s -o /dev/null -w "%{http_code}\n" \
  "https://whirlpool.scoptanalytics.in/api/admin/users" \
  -H "Authorization: Bearer $TAMPERED"
# 401 Unauthorized   ← signature no longer matches payload → REJECTED
# Body: {"error":"invalid token signature"}

7.4 Step 4 — Attack B: algorithm confusion (alg:none)

Strip the signature and declare the token unsigned, a classic JWT bypass against naive verifiers. 

# Set alg:none and drop the signature entirely
NONE_HDR=$(echo -n '{"alg":"none","typ":"JWT"}' | base64 | tr -d '=' | tr '/+' '_-')
PAYLOAD=$(echo -n '{"sub":"8s..","role":"admin","exp":1764587200}' \
          | base64 | tr -d '=' | tr '/+' '_-')
NONE_TOKEN="$NONE_HDR.$PAYLOAD."        # note trailing dot, empty signature

curl -s -o /dev/null -w "%{http_code}\n" \
  "https://whirlpool.scoptanalytics.in/api/admin/users" \
  -H "Authorization: Bearer $NONE_TOKEN"
# 401 Unauthorized   ← server refuses 'none' algorithm → REJECTED
# Body: {"error":"unsupported / invalid algorithm"}

7.5 Step 5 — Attack C: flipped signature byte

# Corrupt a single character of the signature segment
BADSIG_TOKEN="$(echo "$TOKEN" | cut -d. -f1,2).AAAAdeadbeefAAAA"

curl -s -o /dev/null -w "%{http_code}\n" \
  "https://whirlpool.scoptanalytics.in/api/me" \
  -H "Authorization: Bearer $BADSIG_TOKEN"
# 401 Unauthorized   ← HMAC verification fails → REJECTED

7.6 Step 6 — Attack D: expired token (24h window enforcement)

# Replay a token whose exp is in the past
curl -s -o /dev/null -w "%{http_code}\n" \
  "https://whirlpool.scoptanalytics.in/api/me" \
  -H "Authorization: Bearer $EXPIRED_TOKEN"
# 401 Unauthorized   ← {"error":"token expired"} → REJECTED

B Algorithm confusion (alg:none) Reject 401 — invalid algorithm PASS ✔
C Corrupted signature Reject 401 — verification failed PASS ✔

Conclusion — Authentication Integrity Confirmed

The server performs full server-side signature verification on every request, rejects algorithm downgrade, honours the 24-hour expiry, and does not trust client-supplied claims. JWT handling is implemented securely; no authentication bypass or privilege-escalation path was found.

8. Additional Security Validations & Developer Assurances

Beyond the dynamic scan and JWT testing, the following supply-chain, static-analysis and design-level controls were reviewed and corroborated with the Scopt Analytics development team. Each materially reduces the residual risk of the informational scanner items.

Area Validation Result

8.1 Software Composition & Static Analysis

Developer team take: pnpm audit returns 0 vulnerabilities across the dependency tree. Static analysis of the source reveals no hard-coded secrets and no encoding keys. Dependencies are kept current as part of the normal release process.

This addresses the supply-chain and secrets-exposure risk classes (OWASP Top 10:2025 — A02 Cryptographic Failures, A06 Vulnerable & Outdated Components). No vulnerable or outdated component was flagged.

8.2 API Surface, Specification & Documentation

Developer team take: The OpenAPI specification and its Scalar documentation are only viewable by an admin user holding a proper access token. They are not publicly accessible to anonymous or standard users. The Scalar viewer is loaded from CDNJS (a public, well-known GitHub library) purely for the admin to read the docs — it has no path to reach an end user, so the practical opportunity to inject XSS/script through it is extremely limited.

Because the API contract is not exposed, an attacker cannot trivially enumerate endpoints or parameters from a published spec (OWASP Top 10:2025 — A01 Broken Access Control, A05 Security Misconfiguration).

8.3 Sensitive Data Handling & Enumeration Resistance

Developer team take: All API keys and tokens returned over REST are masked. Every record is addressed by a UUID rather than a sequential integer, so enumeration / IDOR-style attacks are not feasible — an attacker cannot guess or iterate identifiers to harvest other records.

UUID-based references combined with server-side JWT authorisation (Section 7) close the common Broken-Access-Control / IDOR vector (OWASP Top 10:2025 — A01).

10. Conclusion

The Whirlpool inspection & analytics platform (whirlpool.scoptanalytics.in) presents a strong security posture with an overall residual risk rating of LOW.

The automated scan surfaced a single Medium item alongside informational notes. Following hands-on Blue Team validation, every material item was confirmed to be a false positive driven by the application's single-page-application architecture, or an accepted informational disclosure that is further mitigated once the production build is promoted behind the corporate VPN:

  • The CSP "unsafe" verdict reflects an intentional, presentation-only inline-style allowance for the ShadCN design system; the policy still denies all remote script execution, structurally resisting XSS and clickjacking.
  • The "GET on all routes" finding is an artefact of the SPA catch-all that returns the same HTML shell for any path — no dangerous HTTP methods exist.
  • Technology fingerprinting exposes no versions; in production it is reachable only by authenticated VPN users.

Critically, authentication integrity was proven by manual testing: signed 24-hour JWTs with server-side verification rejected every tampering, algorithm-confusion, corruption and expiry-replay attempt (Section 7). No authentication bypass or privilege escalation was achievable. These defences are reinforced by the developer assurances in Section 8 — 0 dependency vulnerabilities (pnpm audit), no hard-coded secrets, masked API keys, UUID-based references, and an admin-only OpenAPI/Scalar surface.

Closing Remark — Remediation Posture

Should any future finding be reported against a third-party library, the Scopt Analytics team will live-patch by upgrading the affected library, or by patching the library directly, as part of their standard release and maintenance process — keeping the dependency surface current and minimising the window of exposure.

Scope & Intent of This Assessment

The sole purpose of this exercise was to obtain an understanding of the system, review the deployed code, and exercise developer-level payload manipulation (e.g. JWT tampering).

Final Assessment

No exploitable vulnerabilities were identified within the engagement scope. The remaining recommendations are optional, defence-in-depth hardening. The application is considered fit for continued operation, with residual risk further reduced once promoted to the VPN-gated production environment, subject to ongoing periodic assessment.
